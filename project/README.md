# VanguardRisk-AI: Intelligent Risk, Threat, and Compliance Control Mapping Engine
#### Video Demo: https://youtu.be/YOUR_VIDEO_DYNAMIC_URL
#### Description:

VanguardRisk-AI is a web-based, AI-driven cybersecurity risk, threat, and vulnerability management platform designed to help enterprise security teams organize infrastructure inventories, triage vulnerability discoveries, and automatically align mitigation efforts with global industry standards (specifically NIST SP 800-53 and ISO/IEC 27001).

In traditional corporate governance environments, mapping an engineering vulnerability to a compliance control framework requires manual audits, extensive spreadsheet tracking, and costly consultation hours. This application solves that problem by serving as an intelligent middleware layer. It ingests technical threat descriptions, parses the underlying structural execution pathways, and uses a Large Language Model (LLM) processing layer to map exact, actionable architectural controls directly back to technical operators via an interactive, database-backed checklist dashboard.

---

## Application Architecture & Data Flow

The platform is designed around a relational tiering system that isolates data spaces cleanly while maintaining strict relational boundaries. The architecture relies on an asynchronous Model-View-Controller (MVC) pattern driven by Flask, backed by SQLite, and managed dynamically on the client side using the JavaScript Fetch API.

### Core Database Design (`schema.sql`)
The structural integrity of VanguardRisk-AI relies on a four-tiered hierarchical cascade configuration within SQLite:
1. **`users` Table**: Handles cryptographic authentication records using standard SHA-256 password hashing.
2. **`assets` Table**: Houses enterprise target environments (e.g., Cloud Clusters, Databases, API Gateways) tied back directly to an explicit user account.
3. **`vulnerabilities` Table**: Stores distinct threat profiles, vulnerability descriptions, qualitative severity rankings, and numeric risk indexing targets. Each record points strictly to a unique asset.
4. **`implemented_controls` Table**: Tracks granular remediation actions generated by the AI engine. Each vulnerability dynamically references multiple controls, each containing framework tags (e.g., NIST SI-10) and targeted remediation code or configuration steps.

By structuring the schema with `ON DELETE CASCADE` parameters, deleting an asset safely removes all dependent threat logs and control lists automatically, eliminating orphaned data and preventing data leaks.

## File Infrastructure Breakdown

To keep the application modular and easily maintainable, the workspace code is split cleanly across specialized roles:

* **`app.py`**: The central execution hub of the web service. It manages state transitions, restricts unauthenticated clients via routing decorators, validates form data server-side, and parses structural database modifications.
* **`helpers.py`**: The application's backend toolkit. Beyond standard login enforcement, it houses `call_ai_risk_engine()`. This function interacts with the upstream LLM API and uses strict system-prompting constraints to guarantee that the external AI skips conversational pleasantries and returns structured, machine-ready JSON objects.
* **`static/dashboard.js`**: The client-side interaction engine. It listens for click events on the compliance checklist checkboxes and instantly fires background AJAX requests to the server. This allows security operators to check off mitigated items seamlessly without reloading the entire UI.
* **`templates/layout.html`**: The unified UI shell. It configures the global navbar, manages Bootstrap responsive styling headers, and handles live user feedback via Flask flash notification banners.
* **`templates/index.html`**: The main portfolio landing view. It aggregates mathematical operations on the database to present high-level metrics, such as total monitored assets, overall active threat counters, and the enterprise's total Average Risk Index.
* **`templates/assess.html`**: The ingestion terminal. It provides form layouts, explicit dropdown controls, and radio buttons to capture data fields cleanly for downstream AI processing.
* **`templates/view_vuln.html`**: The tactical workspace. It displays the AI-generated vulnerability summary alongside the live control implementation table, giving engineers clear steps to secure their applications.

## Key Design Choices & Trade-offs

### 1. Enforcing Structural JSON Outputs from the AI Layer
One of the toughest challenges during development was ensuring the LLM consistently returned clean, valid data structures. If an AI model outputs standard conversational text or wraps its response in markdown blocks (like ` ```json `), Python's native `json.loads()` will throw an exception and crash the application thread.

To solve this, a strict defensive system prompt was engineered within `helpers.py`. It explicitly instructs the model to act as a data serialization pipe rather than a chatbot. For resilience, a try-except fallback mechanism was added. If the API times out or returns malformed data, the app falls back to a safe, deterministic local template structure—ensuring the application remains completely functional and crash-proof.

### 2. High-Performance Dashboard Agility via AJAX Toggling
When evaluating how engineers interact with compliance control checklists, a full-page reload on every checkbox click felt sluggish and disruptive to the workflow. To fix this, an asynchronous route (`/control/toggle`) was built to handle small, targeted updates in the background.

When a user toggles a checkbox, `dashboard.js` intercepts the action, fades the row's opacity to indicate an active state, and updates the database via a quick background HTTP POST request. This decoupled architecture significantly cuts down on database load, minimizes network bandwidth, and delivers a fast, responsive user experience.
